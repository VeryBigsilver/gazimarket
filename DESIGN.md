# 1. 요구사항 도출

본 프로젝트는 사용자가 안전하게 중고 물품을 거래할 수 있는 웹 기반 중고거래 플랫폼을 개발하는 것을 목표로 한다. 사용자는 회원가입 후 상품을 등록하고 검색할 수 있으며, 다른 사용자와 채팅 및 송금을 통해 거래를 진행할 수 있다. 또한 관리자는 신고 기능을 통해 악성 사용자와 부적절한 상품을 관리하여 안전한 거래 환경을 제공한다.
## 기능 요구사항
### 1. 회원 관리
* 사용자는 회원가입 및 로그인을 할 수 있어야 한다.
* 사용자는 이름과 이메일을 통해 아이디를 찾을 수 있어야 한다.
* 사용자는 아이디, 이름, 이메일 확인 후 비밀번호를 재설정할 수 있어야 한다.
* 사용자는 자신의 회원 정보를 조회 및 수정할 수 있어야 한다.
### 2. 상품 관리
* 사용자는 상품을 등록할 수 있어야 한다.
* 등록한 상품을 수정하거나 삭제할 수 있어야 한다.
* 모든 사용자는 등록된 상품을 검색하고 조회할 수 있어야 한다.
### 3. 거래 기능
* 사용자 간 송금을 수행할 수 있어야 한다.
* 거래 완료 여부를 확인할 수 있어야 한다.
### 4. 사용자 소통
* 사용자 간 1:1 채팅을 할 수 있어야 한다.
* 전체 사용자가 참여할 수 있는 공용 채팅 기능을 제공해야 한다.
### 5. 신고 기능
* 사용자는 악성 사용자 또는 부적절한 상품을 신고할 수 있어야 한다.
* 신고 시 신고 사유를 입력할 수 있어야 한다.
### 6. 관리자 기능
* 관리자는 회원 정보를 관리할 수 있어야 한다.
* 관리자는 등록된 상품을 관리할 수 있어야 한다.
* 관리자는 신고 내역을 확인할 수 있어야 한다.
* 관리자는 신고가 누적된 상품을 삭제할 수 있어야 한다.
* 관리자는 신고가 누적된 사용자를 휴면 또는 이용 제한 상태로 변경할 수 있어야 한다.
* 관리자는 휴면 또는 이용 제한 사용자를 정상 상태로 복구할 수 있어야 한다.

## 비기능 요구사항
* 사용자 개인정보는 안전하게 보호되어야 한다.
* 비밀번호는 암호화하여 저장해야 한다.
* 웹 서비스는 안정적으로 동작해야 한다.
* 사용자 입력에 대한 검증을 수행하여 보안 취약점을 방지해야 한다.
* 다수의 사용자가 동시에 서비스를 이용할 수 있어야 한다.

# 2. 시스템 설계
## 2.1 회원 관리
플랫폼 이용을 위해 회원가입과 로그인 기능을 제공한다.
* 로그인 페이지
  * 아이디와 비밀번호를 입력하여 로그인
  * 아이디 찾기 및 비밀번호 찾기 페이지로 이동 가능
  * 회원가입 페이지로 이동 가능
* 회원가입 페이지
  * 아이디, 비밀번호, 이름, 이메일을 입력하여 회원가입
  * 아이디 중복 여부 확인
* 아이디 찾기 페이지
  * 이름과 이메일을 입력하여 가입된 아이디 확인
* 비밀번호 찾기 페이지
  * 아이디, 이름, 이메일을 입력하여 본인 정보 확인
  * 새 비밀번호를 입력하여 비밀번호 재설정
* 마이페이지
  * 사용자 프로필 조회
  * 소개글 수정
  * 비밀번호 변경
  * 회원 탈퇴 및 개인정보 비식별 처리
  * 내가 등록한 상품 조회
  * 거래 내역 조회
회원 정보는 데이터베이스에 저장하여 관리한다.
## 2.2 상품 관리
사용자는 상품을 등록하고 조회할 수 있다.
* 메인 페이지
  * 등록된 상품 목록 출력
  * 상품 검색 기능 제공
* 상품 등록 페이지
  * 상품명
  * 가격
  * 상품 설명
  * 상품 사진 업로드
* 상품 상세 페이지
  * 상품 사진
  * 상품명
  * 가격
  * 상품 설명
  * 판매자 정보
  * 채팅 버튼
  * 신고 버튼
* 상품 관리 페이지
  * 내가 등록한 상품 조회
  * 상품 수정
  * 상품 삭제
상품 정보는 데이터베이스에 저장하여 관리한다.
## 2.3 거래 기능
사용자는 플랫폼 내에서 다른 사용자와 거래를 진행할 수 있다.
* 사용자 간 가상 포인트 송금
* 거래 완료 처리
* 거래 내역 조회
* 구매 처리는 데이터베이스 트랜잭션과 쓰기 락을 사용하여 중복 구매와 포인트 불일치를 방지
## 2.4 사용자 소통
거래를 원활하게 진행하기 위해 채팅 기능을 제공한다.
* 전체 채팅
  * 모든 사용자가 참여 가능한 공개 채팅
* 1:1 채팅
  * 판매자와 구매자가 개별적으로 대화 가능
채팅 내역은 데이터베이스에 저장한다.
## 2.5 보안 설계

서비스 이용 과정에서 회원 정보, 비밀번호, 거래 정보 등 민감한 데이터가 처리되므로 다음과 같은 보안 설계를 적용한다.

### 민감정보 보호
* 사용자의 비밀번호는 평문으로 저장하지 않는다.
* SHA-256, bcrypt, Argon2, Werkzeug password hash와 같은 단방향 해시 알고리즘을 사용하여 암호화된 해시 값으로 저장한다.
* 로그인 시 입력된 비밀번호를 다시 해시 검증하여 저장된 `password_hash`와 비교한다.
* 로그 또는 관리자 화면에 비밀번호, 세션 값, 인증 토큰 등 민감정보가 노출되지 않도록 한다.

### 입력 데이터 검증
* 회원가입, 로그인, 상품 등록, 채팅, 신고 등 모든 사용자 입력값에 대해 길이, 필수값, 형식 검증을 수행한다.
* SQL Injection 방지를 위해 문자열 결합 방식의 SQL을 사용하지 않고 Prepared Statement 방식의 파라미터 바인딩을 사용한다.
* 상품 이미지 업로드는 확장자, MIME 타입, 실제 이미지 디코딩 결과를 함께 검증한다.
* 업로드 이미지는 서버에서 재인코딩한 결과만 저장하여 이미지가 아닌 payload 저장을 방지한다.
* 특수문자, 스크립트 태그, 비정상적으로 긴 입력값 등 위험 입력은 blacklist 및 allowlist 기준으로 검증한다.

### XSS 및 CSRF 대응
* XSS 방지를 위해 템플릿 출력 시 Jinja2의 자동 escaping 기능을 사용한다.
* 사용자 입력값은 HTML에 직접 삽입하지 않고 템플릿 엔진을 통해 출력한다.
* 데이터 생성, 수정, 삭제 요청에는 CSRF Token을 적용하여 요청 위조를 방지한다.
* CSRF Token은 사용자 세션과 연결하여 검증하고, 유효하지 않은 요청은 처리하지 않는다.

### 인증 및 권한 관리
* 로그인 성공 시 서버 세션에 사용자 ID를 저장하여 인증 상태를 관리한다.
* 로그인이 필요한 기능은 `login_required`와 같은 인증 검사를 적용한다.
* 관리자 기능은 일반 사용자와 권한을 분리하고, 관리자 계정만 접근할 수 있도록 `admin_required` 검사를 적용한다.
* 이용 제한 또는 휴면 상태인 계정은 로그인과 주요 기능 사용을 차단한다.

### 시간 및 상태 처리
* 세션 만료 시간을 적용하여 장시간 미사용 계정의 인증 상태를 자동 해제한다.
* 인증 토큰 또는 세션성 토큰에는 최대 7일의 expire time을 적용한다.
* 세션 쿠키에는 HttpOnly, SameSite=Lax, Secure 속성을 적용하여 클라이언트 스크립트 접근과 평문 전송 위험을 줄인다.
* 반복 로그인 실패 횟수를 기록하고, 10회 실패 이후에는 다음 로그인 시도까지 대기 시간을 적용한다.
* 대기 시간은 10회 실패 시 30초, 이후 추가 실패 시 1분, 5분 순서로 증가시켜 브루트포스 공격을 방지한다.
* 아이디 찾기와 비밀번호 재설정은 이메일 인증번호 확인 후 처리한다.
* 비밀번호 재설정 요청도 반복 시도를 제한하여 계정 탈취 시도를 줄인다.

### 에러 처리 및 로그 관리
* 예외 메시지와 stack trace를 사용자 화면에 raw 형태로 출력하지 않는다.
* 사용자에게는 일반화된 오류 메시지를 제공하고, 상세 오류는 관리자용 로그에만 기록한다.
* 관리자용 로그는 주기적으로 백업한다.
* 로그에는 비밀번호, 세션 ID, 인증 토큰, 개인정보 원문 등 민감정보를 저장하지 않는다.

### 캡슐화 적용
* 데이터베이스 연결, 로그인 검사, 관리자 권한 검사, 파일 저장, 입력 변환 기능은 별도 함수로 분리하여 관리한다.
* 인증과 권한 검증 로직을 라우트마다 직접 중복 작성하지 않고 데코레이터나 헬퍼 함수로 캡슐화한다.
* 보안 관련 정책이 변경될 경우 공통 함수만 수정해도 전체 기능에 일관되게 반영되도록 설계한다.

# 3. 도입 기술 스택

## 3.1 프론트엔드
* 언어: HTML, CSS
* 템플릿 엔진: Jinja2
* 화면 구성: Flask 서버에서 렌더링한 HTML 페이지를 웹 브라우저에 전달하는 서버 사이드 렌더링 방식
* 정적 파일: `static/style.css`, 상품 이미지 업로드 파일, 기본 SVG 이미지 사용

## 3.2 백엔드
* 언어: Python 3.12
* 웹 프레임워크: Flask 3
* 주요 라이브러리
  * Werkzeug: 비밀번호 해시 생성 및 검증, 업로드 파일명 정리
  * PyYAML: YAML 설정 파일 로드
  * sqlite3: SQLite 데이터베이스 연결
* 인증 방식: Flask session 기반 로그인 상태 관리

## 3.3 데이터베이스
* 데이터베이스: SQLite
* 저장 위치: `instance/gazimarket.sqlite3`
* 사용 목적: 회원, 상품, 거래, 채팅, 신고 데이터를 영구 저장

# 4. 시스템 아키텍처 구성

가지마켓은 웹 브라우저, Flask 웹 서버, SQLite 데이터베이스로 구성된다.

1. 사용자는 웹 브라우저에서 가지마켓 페이지에 접속한다.
2. 웹 브라우저는 HTTP 요청을 Flask 웹 서버로 전송한다.
3. Flask 웹 서버는 요청 URL과 HTTP 메서드에 따라 적절한 라우트 함수를 실행한다.
4. 회원가입, 로그인, 상품 조회, 거래, 채팅, 신고 등 데이터 처리가 필요한 경우 Flask 서버가 SQLite 데이터베이스에 SQL 쿼리를 실행한다.
5. Flask 서버는 조회 또는 처리 결과를 Jinja2 템플릿에 전달하여 HTML을 생성한다.
6. 생성된 HTML, CSS, 이미지 파일이 웹 브라우저로 응답되어 사용자 화면에 표시된다.

구조 요약:

```text
웹 브라우저
  -> HTTP 요청
Flask 웹 서버(app.py)
  -> SQL 쿼리
SQLite 데이터베이스(instance/gazimarket.sqlite3)
  -> 조회/처리 결과
Flask 웹 서버
  -> HTML 응답
웹 브라우저
```

업로드된 상품 이미지는 데이터베이스에 직접 저장하지 않고 `static/uploads/` 디렉터리에 파일로 저장한다. 데이터베이스에는 해당 이미지 파일 경로만 저장한다.

# 5. 데이터베이스 설계

## 5.1 users 테이블

회원 정보를 저장한다. 회원 탈퇴 시 row는 삭제하지 않고 거래, 상품, 채팅, 신고 내역의 참조 관계를 유지한다. 대신 개인정보를 비식별 값으로 변경하고 `status`를 `deleted`로 저장한다.

| 컬럼명 | 자료형 | 설명 |
| --- | --- | --- |
| id | INTEGER | 회원 고유 ID, 기본 키 |
| username | TEXT | 로그인 아이디, 중복 불가 |
| password_hash | TEXT | 해시 처리된 비밀번호 |
| name | TEXT | 사용자 이름 |
| email | TEXT | 이메일 주소 |
| bio | TEXT | 자기소개 |
| points | INTEGER | 보유 가상 포인트 |
| role | TEXT | 권한, 일반 사용자 또는 관리자 |
| status | TEXT | 계정 상태, active/restricted/dormant/deleted |
| created_at | TEXT | 가입일 |

탈퇴 처리 시 주요 비식별화 값:

| 항목 | 처리 방식 |
| --- | --- |
| username | `deleted_user_<id>` |
| password_hash | 랜덤 비밀번호의 해시 값 |
| name | `탈퇴한 사용자` |
| email | `deleted_user_<id>@deleted.local` |
| bio | 빈 문자열 |
| points | 0 |
| status | `deleted` |

## 5.2 products 테이블

상품 정보를 저장한다.

# 6. API 명세

현재 가지마켓은 별도의 JSON API 서버를 분리하지 않고, Flask 라우트가 HTML 페이지 렌더링과 폼 처리를 함께 담당한다. 웹 브라우저는 RESTful 스타일의 URL에 HTTP GET 또는 POST 요청을 보내며, 서버는 HTML 응답 또는 리다이렉트로 처리 결과를 반환한다.

## 6.1 주요 라우트

| 기능 | HTTP 메서드 | URL | 설명 |
| --- | --- | --- | --- |
| 상품 목록 | GET | `/` | 상품 목록 조회, 검색, 정렬. 판매 완료 상품은 항상 하단 표시 |
| 상품 목록 갱신 | GET | `/products/list` | 검색어와 정렬 기준에 맞는 상품 목록 HTML 조회, 자동 갱신용 |
| 회원가입 | GET, POST | `/register` | 회원가입 화면 표시 및 가입 처리 |
| 로그인 | GET, POST | `/login` | 로그인 화면 표시 및 로그인 처리, 반복 실패 시 지연 적용 |
| 로그아웃 | POST | `/logout` | CSRF 검증 후 세션 초기화 |
| 아이디 찾기 | GET, POST | `/find-id` | 이름과 이메일 확인 후 이메일 인증번호 검증, 아이디 조회 |
| 비밀번호 찾기 | GET, POST | `/reset-password` | 본인 정보 확인 후 이메일 인증번호 검증, 비밀번호 재설정 |
| 마이페이지 | GET, POST | `/mypage` | 프로필 수정, 비밀번호 변경, 상품/거래 내역 조회, 회원 탈퇴 및 개인정보 비식별화 |
| 상품 등록 | GET, POST | `/products/new` | 상품 등록 화면 표시 및 등록 처리 |
| 상품 상세 | GET | `/products/<product_id>` | 상품 상세 조회 |
| 상품 수정 | GET, POST | `/products/<product_id>/edit` | 상품 수정 화면 표시 및 수정 처리 |
| 상품 삭제 | POST | `/products/<product_id>/delete` | 상품 삭제 |
| 상품 구매 | POST | `/products/<product_id>/buy` | 가상 포인트 송금 및 거래 생성 |
| 거래 완료 | POST | `/transactions/<transaction_id>/complete` | 거래 완료 처리 |
| 전체 채팅 | GET, POST | `/chat/public` | 전체 채팅 조회 및 메시지 등록 |
| 전체 채팅 메시지 조회 | GET | `/chat/public/messages` | `since_id` 이후 새 메시지 JSON 조회 |
| 1:1 채팅 목록 | GET | `/chat/private` | 내가 참여한 1:1 채팅방과 미확인 메시지 조회 |
| 1:1 채팅 | GET, POST | `/chat/product/<product_id>/<seller_id>` | 상품 기준 상대 사용자와 1:1 채팅 |
| 1:1 채팅 메시지 조회 | GET | `/chat/product/<product_id>/<seller_id>/messages` | `since_id` 이후 새 1:1 메시지 JSON 조회 |
| 신고 | GET, POST | `/reports/new` | 신고 화면 표시 및 신고 접수. 자기 자신과 자기 상품 신고는 차단 |
| 관리자 | GET, POST | `/admin` | 회원 상태 관리, 휴면/제한/정상 복구, 상품 차단/삭제, 신고 처리 |

## 6.2 요청 및 응답 규칙

* 조회 화면은 주로 GET 요청을 사용한다.
* 데이터 생성, 수정, 삭제 작업은 POST 요청을 사용한다.
* 모든 POST 요청은 CSRF Token을 함께 전송해야 한다.
* 폼 입력 오류가 있으면 `flash` 메시지로 오류 내용을 표시한다.
* 처리가 완료되면 중복 제출을 줄이기 위해 관련 페이지로 리다이렉트한다.
* 로그인 여부가 필요한 기능은 세션을 확인하고, 비로그인 사용자는 로그인 페이지로 이동시킨다.
* `/products/list`, `/chat/public/messages`, `/chat/product/<product_id>/<seller_id>/messages`는 자동 화면 갱신을 위해 JSON 응답을 반환한다.
